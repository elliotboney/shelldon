## What went well

1. **The Epic 5 "build the mechanism ahead of the consumer" bet paid off completely.** 6.2's dream needed `prompt_builder` (5.4), the `cost` weight (5.2 — literally specced "a future dream declares `cost=3`"), `history_owner_text` (5.4), and the 5.2/5.3 gates — **all pre-built and tested before their user existed.** The dream landed as one job registration + one builder + one dispatch branch, with **zero new turn-lifecycle code**. The Epic 5 retro's exact prediction ("Epic 6's dream job IS a 5.4 turn job") held.
2. **6.1 stayed genuinely thin (3.4-shaped).** One op + one table + one dispatch branch; the worker decoded the new `ProposedOp` union variant **for free** (no `worker.py` change), no `SCHEMA_VERSION` bump (additive). The contract-additivity discipline is now reflexive.
3. **The 6.1 TOCTOU decision was a model "fix-now" call.** Review flagged the non-atomic SELECT+UPDATE dedup as a race the moment 6.2 becomes a second writer. The owner chose to fix it then — with an atomic UPSERT + unique partial index that was **net-simpler** than the original code. Fixing the race *ahead of* its consumer is the Epic 5 lesson applied to data integrity.
4. **runtime.py coupling discipline was consciously applied** — even for the biggest story, 6.2's runtime change was held to one job + one builder + one branch (no lifecycle refactor), exactly as the Epic 5 retro asked.
5. **The firehose/keepsake split held.** Learnings stage in sqlite (dedup/count/status lifecycle); the dream promotes the durable signal up to markdown. No confusion between the two stores across either story.
6. **0 incidents, 0 deps, contracts KEPT, +37 tests** — the clean-delivery streak (every epic since 1) continues.

## Challenges / themes (systemic, no blame)

1. **THE DOMINANT PROJECT RISK IS NOW EXPLICIT: the entire memory + learning + autonomy line is mechanism-proven and NEVER run against a live LLM.** Epic 6 is the sharpest case — the dream's whole *value* is the LLM's judgment ("which learnings are durable?"), and that judgment is 100% untested (synthetic `Result`s only). The dream directive wording, the `SYSTEM_INSTRUCTION` op vocabulary, the summary injection, whether the model even *emits* `capture_learning`/`resolve_learning` — all unverified against a real model. This risk has been carried since Epic 5; after Epic 6 it is the **largest unknown in the project.**
2. **A real surfacing GAP was discovered mid-build: `facts/`/`people/` are NOT injected into prompts.** 4.4 assembly injects `about.md` + `summary.md` + history recall — but not the `facts/`/`people/` trees. So a `remember`→`facts/` promotion is durable storage that **never shapes a future reply.** CAP-11 only works via `about.md`/`summary.md`; the dream had to be steered to `rewrite_about` as a workaround. A whole class of "durable knowledge" is currently write-only. Caught in dev (good), but it undercuts the memory story's completeness.
3. **6.2 drew 6 review patches — the most of any recent feature story.** None were deep bugs (directive steering, a `log.warning` that would spam prod, two missing tests, two inaccurate comments) — but the biggest story accumulated the most polish-debt, and two patches were the *same* input-edge/accuracy class (newline-in-observation, comment drift) the Epic 5 checklist targets. The checklist helps; the largest story still collects them.
4. **The `runtime.py` extract trigger is now HIT.** The Epic 5 icebox said "extract the scheduler driver + dispatch when Epic 6's dream grows `runtime.py` again." It did — `Core` now hosts reflex + checkpoint + proactive + dream jobs, **three** prompt builders, the dispatch gate, and the turn lifecycle. The extract is no longer hypothetical; it's due before Epic 7 (plugin events will also touch core).
5. **Two `Idle` turn jobs (proactive 1h, dream 6h) coexist with a known cold-start collision** (both due in one tick → one defers and waits a full fresh idle stretch). Acceptable, noted in-code, not fixed — a latent rough edge.

## 🚨 Significant-discovery check → Epic 7 (and the real fork)

**Epic 7 (Extensibility & Optional Embodiment) is EXPLICITLY OPTIONAL — the core pet is feature-complete after Epic 6.** Epic 7 (plugin-host, broadcast events, XP plugin, physical sensing) depends on the bus/contracts (done since Epic 1) and the AD-8 plugin seam; **nothing in Epic 6 changes its plan or blocks it.** No re-plan needed.

**But the honest strategic fork is NOT "Epic 7 vs not" — it's "verify vs extend."** The daily-driver + autonomy + memory + learning loop is **built and mechanism-proven, but never run against a live LLM and never deployed.** The two highest-value next moves are *closing that gap*, not adding Epic 7's surface:
- **Live-LLM smoke verification** — does a real model actually drive the turn, emit the ops, and dream usefully? (The dominant risk.)
- **`facts/` surfacing** — without it, a third of the memory model is write-only.

Adding Epic 7 (more plugins, more surface) on top of an unverified core would compound the unknown. The retro's recommendation: **verify the core against a real brain before extending it.**

## Readiness assessment

- **Tests/quality:** 448 green, contracts KEPT, deterministic (synthetic Results, injected clocks). Solid as a *mechanism* — but the LLM-judgment core of the dream is untested (no live lane).
- **Deployment:** pre-deployment; nothing shipped. The whole learn-improve loop is live in-process and parked in the soak.
- **Technical health:** stable; `core/` modules (`history`/`memory`/`budget`/`power`/`proactive`/`scheduler`) stay cleanly pure/LLM-free — but **`core/runtime.py` coupling is now over the line** (action #3). One real architecture gap: `facts/` not surfaced (action #2).
- **Blockers for Epic 7:** none technical. The real gate is strategic — verify the core against a live LLM (action #1) before extending.

**Verdict:** Epic 6 is genuinely complete, and with it the **core pet is FEATURE-complete** — daily-driver + resilient brain + alive personality + memory + autonomy + learning. The next move is the honest one the project has deferred for two epics: **run it against a real brain.** Epic 7 is real, optional enrichment — but verifying and deploying the core should come first.
