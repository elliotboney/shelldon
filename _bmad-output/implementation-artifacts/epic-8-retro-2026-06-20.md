## What went well

1. **The whole architecture paid off on real hardware.** The fork-server's "ephemeral worker, no accumulation" thesis — the load-bearing reason for half the design — held under a hard `MemoryMax=400M` cgroup cap on a 416MB box, through real Telegram turns. The LLM-free core, the transport seam, the renderer seam, and the gated-hardware discipline all did exactly what they were built for.
2. **The seams made each deploy story thin.** Telegram was "a different inbound/outbound" on the 1.6 transport seam (core untouched). E-Ink was "a real Renderer" behind the 1.7 stub seam. The env-gated `_default_renderer`/`_transport_actor` selectors mirror the worker-uid gating. "Ship the seam, gate the hardware" (from 1.5/4.3/7.4) made the Pi bring-up incremental, not a rewrite.
3. **Systematic debugging earned its keep on the memory bug.** Three plausible theories (fork+WAL+ro, sd-card filesystem, read-only-mode/busy_timeout) were each **disproven by a minimal on-device repro** before the real trigger was bisected (inherit core's WAL connection × `closerange`). Guessing would have shipped a wrong "fix." The bug — SQLite is not fork-safe — is a deep one that only manifested in the real fork environment (masked on macOS and by inheriting-process repros).
4. **"Do it hands-on, then document" fit deployment.** 8.1/8.3/8.4 were interactive ops (ssh to the Pi, install, run, observe) documented after they worked — the right shape for deployment vs. story-file-first. The render-to-PNG-and-look-at-it loop (verifying font glyphs off-panel before touching hardware) is a reusable trick.
5. **Verify-before-extend was finally honored.** Epic 7's retro flagged that the core had never run against a live LLM; Epic 8 led with that (8.0) before deploying — and the deployment then surfaced the real memory bug that synthetic tests never could.

## Challenges / lessons

1. **SQLite is not fork-safe — and the bug hid behind fail-soft.** The forked worker inherited core's open WAL connection; `closerange` tore its `-shm`; the read-only history open failed every turn. The 4.4 fail-soft (degrade gracefully) meant it *replied fine* while silently having no short-term memory — a correctness bug that looked like nothing for as long as nobody looked. **Lesson:** fail-soft can mask a real defect indefinitely; a degraded path should be loud enough to notice in production (the log was there, but easy to miss). **Carried risk:** the fork forks from core's process (inheriting its fds) — the docstring's intended "fork-server as its own clean process" (Story 1.8's IPC wiring) was never fully built; the SQLite fix patches the symptom of that. If other not-fork-safe resources get inherited later, the same class recurs.
2. **`MemoryMax=400M` on a 416MB box is tight.** It held (NRestarts=0, settled 279MB), but a 5-process app + a forked worker under one cgroup cap leaves little headroom. Watch for OOM-kills under heavier prompts / longer histories.
3. **Pi install friction:** `lgpio` builds from source and needs `swig` + `liblgpio-dev` (apt) — `setup-pi.sh` handles it, but it's a non-obvious failure if the batch `uv pip install` aborts.
