# Epic 10 Retrospective — Persona-as-Files

**Date:** 2026-06-26
**Participants:** Elliot (project lead), Claude (dev/facilitator) — solo project, no party-mode fiction.
**Epic status at retro:** 5/5 stories dev-complete. 10.1–10.3 `done`; 10.4 + 10.5 `review` (loose end → Action 1).

## What this epic was

Moved every piece of hardcoded LLM-facing prose out of code and into bot- and owner-editable markdown under `~/.shelldon/memory/` — the v1 worktree model rebuilt under v2 invariants. The character (`SYSTEM_INSTRUCTION`, proactive/dream copy) became *data the bot reads each turn*, editable by the bot via the memory-op path (AD-5 single-writer) and by the owner on disk. Foundation (10.1) → bot-write ops + gated DIRECTIVE (10.2) → proactive/dream prose to files + autonomous edit (10.3) → first-run onboarding (10.4) → cost/lazy-load/Pi migration (10.5).

## What went well

1. **Clean "data-not-code" refactor.** Contracts green every story, zero dependency creep, ops added only where a genuine new capability demanded them. The invariants (AD-1 core LLM-free, AD-5 single-writer, AD-6 worker read-only, AD-3 fork no-accumulation) held across all 5 without strain.
2. **Aggressive seam reuse kept stories small.** `_seed_persona` copy-if-absent was reused verbatim 3× (BOT_INSTRUCTIONS → +HEARTBEAT/DREAM → +BOOTSTRAP → +TOOLS/ARCHITECTURE). The owner-gated `DIRECTIVE` rewrite (10.2) reused Story 9.3's park/Approve Telegram plumbing with **zero schema change** — packed as a `(messages=(), ToolCall)` blob.
3. **Golden byte-parity tests guarded every prose move.** Day-one no-op (10.1), proactive byte-identical (10.3), byte-stable cache prefix (10.5) — moving prose to files had to change nothing observable, and the tests enforced it.
4. **Review caught real wire bugs, not nits.** 10.2: the non-greedy ops-block regex closed at a *nested* ```` ```ops ```` inside `rewrite_instructions` content, so the instructions op silently never round-tripped (fixed by anchoring the closing fence to line-start); plus a park-key sqlite INSERT-OR-REPLACE collision when two parking ops landed in one Result. 10.5: the byte-stable-prefix guard + the keyword false-positive precision fix.
5. **Honest in-flight design overrides for correctness.** 10.1 rejected the story's *coarse* shared fail-soft handler for independent per-file `_safe_read` (a corrupt SOUL.md can't drop the system instruction). 10.2 rejected the design's "9.3 verbatim → resumed worker applies DIRECTIVE" because it violates AD-5 — chose core-applies-on-Approve, no worker resume.

## What repeatedly bit us — the one dominant theme

**Fail-soft behavior at file/template boundaries was the entire risk surface, and nearly every review patch.** The mechanism was right on the first pass every time; the *boundary and teardown behavior* held the bugs:

- **10.2:** ops-fence nesting silently dropped a valid op; park collision clobbered a sqlite row.
- **10.3:** missing template slots silently dropped feeling / learnings / mood; `read_heartbeat`/`read_dream` lacked the OSError catch their fail-soft promise implied.
- **10.4:** corruption swallowed without the promised log (**still an OPEN patch** — the AC5 logging gap); a fail-soft test deleted `BOOTSTRAP.md`, but `gather_context` re-seeds absent files on construction, so the delete was silently undone and the test passed green-but-wrong (fixed by using a corrupt non-UTF-8 file, which seeding never overwrites).
- **10.5:** the "untracked / absent seed file" failure class from 10.4 was explicitly *guarded* this time — recurred enough to become a known hazard later stories defend against proactively.
