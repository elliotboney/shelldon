## What went well

1. **A reusable durability idiom emerged and propagated.** 3.1's atomic-write (temp→fsync→`os.replace`) + corruption-tolerant load (absent→seed, corrupt→defaults+warn-never-raise) was reused verbatim by 3.3 for `faces.toml`. The team is building idioms, not re-solving.
2. **Lock-free single-writer held under real concurrency.** Synchronous `apply_patch` on the one core loop delivered reflex/turn coexistence (3.2 AC2) with no locks — the architecture's serial-access bet paid off in practice.
3. **Pre-empting prior review findings cut repeat cycles.** 3.2 built in all three of 3.1's review fixes (positive-interval guard, loop-survives-a-transient-error, long-lived task off `_bg`) *before* review — so its review found nothing in that class.
4. **Owner-in-the-loop caught a cross-epic fork before any code.** 3.3's create-story surfaced faces-as-data vs enum, core-maps vs display-maps, and the 3.4 split by reading the turn path. The result: faces became self-modifiable data, and Epic 4's memory-op protocol was *not* prematurely built under Epic 3.

## Challenges / recurring themes (no blame — systemic)

1. **"Never write real `$HOME`" was caught in verify twice** (3.2's `last_interaction` dirtying state on every inbound; 3.3's faces seed-on-load). Both fixes were reactive conftest patches. The isolation chased the bug instead of preceding it.
2. **The 1.9 soak's `_seq`/`_bg` invariants are brittle to new background emitters.** 3.2's reflex task and 3.3's mood-face push each forced a harness tweak (`reflex_interval=3600`). The strict invariants are valuable but undocumented for future background work.
3. **The same review nitpicks recur across stories** — input/range guards, no-silent-swallow (log on fallback), no WHAT-comments, share test helpers via conftest, value-not-truthiness asserts. Five stories of the same feedback class.

## 🚨 Significant discovery — affects Epic 4

The worker is **fire-and-forget**; the **broker** emits `Result → core`, so the worker never sees the completion. Epic 4.2's "worker proposes memory-ops" (and the deferred 3.4) **requires reshaping that topology** — the worker must receive the completion and emit a `Result` carrying structured proposed-ops, OR the op-extraction must live in broker/core. **This is an up-front Epic 4 planning decision, not a mid-story one.** (Owner decision: resolve it as the first task of Story 4.2; 3.4 is blocked on it.)
